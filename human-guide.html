Agent Workflow Starter User Guide

A practical guide for using Claude, Claude Code, GitHub, templates, checks, and lightweight setup/config to run reviewable agent workflows.

What this guide is for

Agent Workflow Starter helps a user turn knowledge-work goals into scoped, reviewable, reusable work. It is for writing, research, guides, tutorials, course material, websites, digital products, project docs, and lightweight supporting tools.

The user does not need to become a software developer. Claude handles planning, setup proposals, repo mechanics, checks, and handoffs while the user owns judgment, priorities, review, publishing, and merge decisions.

First-run setup

After cloning or forking the repo, ask Claude Code to set it up. The user should not manually perform setup unless Claude Code asks for approval or a system-level action.

You are setting up my cloned Agent Workflow Starter repo.
Read README.md, human-guide.html, agent-ops.md, and bootstrap/bootstrap.md.
Inspect the current state before changing files.
Prepare only what is needed for the repo to work locally.
Run the relevant checks.
Do not change dotfiles, hooks, GitHub workflows, publishing settings, or account-level settings without approval.
End with a setup report: what exists, what changed, checks run, what I should review, and suggested next steps.

The workflow spine

Goal → Plan Doc → Plan Review → Implementation Plan → Definition of Done → Todos → GitHub Issue → Claude Code Handoff → Branch → Execute → Verify → Pull Request → Review → Merge → Version / Release Notes → Final Handoff

Use this sequence for non-trivial work. For tiny edits, Claude can compress the sequence, but it should still preserve scope, verification, and handoff.

Why use a plan doc before execution

A plan doc prevents vague requests from turning into broad edits. It gives Claude a bounded task, a definition of done, known risks, and explicit non-goals.

A useful plan doc includes the goal, non-goals, current state, assumptions, open questions, risks, implementation plan, files likely touched, verification plan, stop conditions, and follow-up work that should not be done now.

How to iterate the plan doc

Before execution, ask Claude to review the plan for missing assumptions, scope creep, unclear dependencies, risky edits, and work that should be split into separate issues.

Review this plan before execution.
What is missing?
What could break?
What is too broad?
What depends on something else?
What should be deferred?
What requires approval?
What is the smallest useful version?

Canonical ordering

Canonical ordering means deciding the default sequence before execution. Use this order unless the issue says otherwise:

  1. confirm the goal
  2. inspect current state
  3. identify dependencies
  4. set up required config
  5. define the smallest useful unit
  6. execute that unit
  7. verify it
  8. record follow-ups instead of expanding scope

Setup flow

Audit current state → Identify dependencies → Set up minimum config → Add templates/checks only if needed → Verify setup → Start scoped work

Examples of setup include GitHub templates, Claude defaults, local scripts, dotfiles, hooks, smoke checks, repo maps, or a simple folder structure. Keep setup small and reversible.

Atomic design and dependencies

Atomic design here means breaking work into small reusable units before building larger workflows. A source note, claim, section, prompt, template, checklist item, lesson step, hook, smoke check, or release note can all be an atom.

Build from atoms to larger parts: atoms become sections, sections become guides, guides become workflows, workflows become releases. Claude should identify dependencies before editing.

Setup/config: dotfiles

Dotfiles are setup/config. They are portable defaults that make the working environment predictable: shell config, Git config, editor defaults, Claude defaults, scripts, aliases, hooks, and local automation.

Treat dotfiles as setup work. Claude may help set them up after approval. Do not turn dotfiles into research notes, tutorials, content, or a second operating system.

Hooks

Hooks are local scripts that run automatically during Git actions. They can prevent predictable mistakes, such as skipping checks or working on the wrong branch. Claude must ask before adding or changing hooks.

CI, lint, and smoke

CheckMeaning
CIAutomated GitHub checks that run before changes merge.

Dotdog

Dotdog can provide repo mapping and lightweight validation when the repo grows. Humans do not need to operate Dotdog during normal use. Claude Code can use it when appropriate.

Advanced tools flow

Repeated friction → Tool proposal → Dependency check → Smallest useful setup → Verification → Documentation → Follow-up issue if more is needed

Advanced tools are still relevant even when the human is not doing hard programming. Add them when they remove repeated friction or make agent work safer.

Token efficiency and preventing drift

Give Claude the issue, plan doc, and relevant files. Ask for diffs or summaries instead of full file dumps. Define non-goals, stop conditions, and follow-up issues before execution.

Approval rules

Claude should ask before changing dotfiles, hooks, GitHub workflows, publishing settings, account-level settings, dependencies, release rules, or broad system instructions.
