Part 1: Basics

Start here. This part is enough for most writing, teaching, research, guide, and documentation work.

Principle

The core rule

AI is useful, but not wise. Treat it like a fast, forgetful assistant: good at motion, bad at owning direction.

Human supplies

Intent, judgment, standards, taste, review, and final decisions.

Agent supplies

Drafting, organizing, searching, comparing, checking, scripting, and momentum.

Repo supplies

Files, issues, pull requests, checks, templates, and handoffs as source of truth.

Do not let the agent carry the goal. Give it small tasks, real files, and checkable outputs.

Prompting

Set clear goals

A useful goal names the artifact, the audience, the scope, the non-goals, and the standard for done.

Create a short guide for [audience] that helps them [outcome].
Use existing files as source material.
Do not add unrelated sections.
End with what changed, what needs review, and what should happen next.

Review

Use AI without outsourcing intent or judgment

Ask the agent to do bounded work: inspect, compare, outline, draft, edit, test, search, summarize, write a small script, or prepare a pull request. Do not ask it to decide what matters without giving it standards.

When the agent makes a decision, question it. Ask why that choice was made, what evidence supports it, what tradeoff it accepted, and what assumption would change the answer.

Why this choice?
What evidence supports it?
What tradeoff did it accept?
What assumption would change the answer?

Context

Context hygiene

Long conversations can become noisy. The agent may carry forward stale assumptions, bad earlier outputs, or old decisions that no longer apply. When bad assumptions stay in the working context, the session can become poisoned.

Do not keep dragging a confused conversation forward. When the context is long, stale, contradictory, or unreliable, stop and write a handoff.

Goal:
Current state:
Decisions:
Files touched:
Commands run:
Checks passing/failing:
Next action:
Open questions:

A new session should start from files, issues, pull requests, checks, and handoffs, not vague chat memory.

Tokens

Token optimization

Tokens are working memory. Do not spend them on whole files, stale chats, or repeated background when a focused source will do.

Bad pattern

Paste everything, ask the agent to remember it, and keep extending the same confused thread.

Better pattern

Search first, read relevant sections, edit from files, use diffs, and write handoffs.

Search first.
Read only relevant files.
Prefer diffs over full dumps.
Use handoffs instead of long chat history.
Use local SQLite search when repeated lookup wastes context.
Keep source files as truth.

Restart

When to start fresh

Start fresh when the current chat is no longer a reliable workspace. Do not restart just because the chat is long. Restart when the chat is steering the work in the wrong direction.

SignalWhat to do
The agent repeats a wrong assumption.Stop. Write the correct current state into a handoff.
The agent edits before inspecting files.Stop. Restart from the repo, issue, PR, and relevant files.
The task changed direction.Stop. Record the old decision as abandoned and state the new goal.
Several plans are mixed together.Stop. Choose one plan and defer the rest.
The answer sounds confident but unsupported.Stop. Ask for evidence or restart from source files.
Restart prompt:
Use this handoff as the source of truth.
Inspect the repo before editing.
Ignore earlier chat unless it is repeated in the handoff.
State the next smallest useful step.

Setup

First-run setup

After cloning or forking the repo, ask Claude Code to inspect and set it up. It should prepare only what is needed and report what changed.

You are setting up my cloned Agent Workflow Starter repo.
Read README.md, CLAUDE.md, human-guide.html, agent-ops.md, and bootstrap/bootstrap.md.
Inspect the current state before changing files.
Prepare only what is needed for the repo to work locally.
Run the relevant checks.
Do not change dotfiles, hooks, GitHub workflows, publishing settings, dependencies, database schema, or account-level settings without approval.
End with a setup report: what exists, what changed, checks run, what I should review, and suggested next steps.

Operating loop

The workflow spine

Goal → Inspect → Plan → Issue → Handoff → Branch → Execute → Verify → Pull Request → Review → Merge → Final Handoff

For tiny edits, compress the sequence. For meaningful work, keep the spine. It prevents vague requests from becoming broad, unreviewable changes.

Automation

Small scripts and mini apps

AI is useful for building small local tools that remove repeated chores. A discrete Python, Go, or Rust script can save more time than another prompt.

Use Python first for quick local automation. Use Go for portable CLI tools. Use Rust only when speed, safety, or single-binary discipline matters.

Use when a task repeats.
Keep it local-first.
Give it one job.
Add one command to run it.
Add one check or smoke test.
Document the input, output, and rollback.

Local setup

macOS Apple Silicon

The likely local setup is macOS on Apple Silicon. Keep it simple: Xcode Command Line Tools, Homebrew, Git, ripgrep, GitHub CLI, GitHub CLI auth, Node/npm, Python 3, Claude Code, and the repo scripts. Do not install extra packages until a project actually needs them.

Run npm run setup:macos to check versions and missing tools. After review, npm run setup:macos:install can install missing Homebrew-managed tools. Xcode Command Line Tools and Homebrew may still require interactive installation.

Local search

Local SQLite knowledge

SQLite local knowledge is optional. Use it when repeated file lookups waste context or when the project needs a reusable local index.

Start with FTS5 keyword search. Keep source files as truth. Store generated database files under a gitignored local path. Add embeddings or vector search later only if keyword search stops being enough.

npm run kb:init
npm run kb:rebuild
npm run kb:search -- "handoff"

Part 3: Advanced

Use this when repeated friction justifies a stronger tool, rule, or workflow. Advanced does not mean complicated. It means optional, specific, and justified.

Decision

Advanced decision rule

Do not add an advanced tool because it sounds useful. Add it when it saves time, reduces drift, improves reviewability, or prevents a mistake that keeps repeating.

Use when:
the friction repeats
the output can be checked
the workflow stays understandable
the user approves any risky change

Do not use when:
the task is one-off
a direct edit is enough
the setup adds more work than it removes
the tool hides judgment the user should keep

Packages

Packages and named tools

Treat named tools as software, not shorthand. A package belongs in the guide only when the repo says what it is, why it is used, how to run it, how to check it, and when not to use it.

NeedRequired note
Package nameUse the real name and link or local docs when available.
Install pathState whether it is Homebrew, npm, Python, Go, Rust, or manual.
CommandShow the exact command the user or agent should run.
CheckShow how to tell whether it worked.
Approval gateAsk before adding dependencies, hooks, paid services, credentials, or publishing.

If a name is not documented as a package, do not define it as a mode in the public guide.

Automation

Automation guardrails

Hooks, browser automation, worktrees, hosting, and UI prototyping are useful only when they make work safer or easier to review.

Ask before:
adding hooks
adding dependencies
changing CI
publishing
using credentials
creating or deleting worktrees
copying generated prototype code into the repo

For each automation, require one clear command, one clear failure mode, and one rollback path.

Safety

Approval gates

The agent should ask before destructive actions, deleting files, publishing, credentials, hooks, broad rewrites, paid services, database/schema changes, dependency additions, GitHub workflow changes, release/versioning changes, system instruction changes, or merging to main.

These are gates, not handcuffs. The agent can still choose reasonable tools and dependencies when the task clearly needs them and the change is reviewable.
